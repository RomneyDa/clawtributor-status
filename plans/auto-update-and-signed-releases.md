# Auto-Update and Signed Releases Plan

## Goal

Ship Clawtributor Status with trustworthy macOS and Windows downloads, then add a quiet, user-controlled auto-update flow.

Do not enable auto-update until production signing is working on both supported platforms.

## Current State

- GitHub Releases publish macOS and Windows artifacts.
- OAuth is privacy-minimal and uses `read:user`.
- App queries are OpenClaw-focused.
- macOS builds are not Developer ID signed or notarized.
- Windows builds are not signed with a publisher certificate.
- Auto-update is not implemented.

## Release Policy

- Stable releases use semver tags like `v0.1.4`.
- Release artifacts are public GitHub Release assets.
- Keep portable ZIPs as fallback downloads.
- Use installable targets for auto-update:
  - macOS: `dmg` plus `zip`
  - Windows: `nsis` plus optional `zip`
- Do not publish updater metadata for untrusted unsigned builds.

## Phase 1: macOS Signing and Notarization

Prerequisites:

- Apple Developer Program membership.
- Developer ID Application certificate.
- App-specific password or App Store Connect API key for notarization.
- Apple Team ID.

GitHub Actions secrets:

- `MAC_CSC_LINK`: base64-encoded `.p12` certificate or secure URL.
- `MAC_CSC_KEY_PASSWORD`: password for the `.p12`.
- `APPLE_ID`: Apple ID used for notarization.
- `APPLE_APP_SPECIFIC_PASSWORD`: app-specific password.
- `APPLE_TEAM_ID`: Apple Developer Team ID.

Implementation:

- Configure electron-builder mac target with hardened runtime enabled.
- Ensure mac builds include both `dmg` and `zip`.
- Let electron-builder sign with `CSC_LINK` / `CSC_KEY_PASSWORD`.
- Let electron-builder notarize with Apple credentials.
- Keep CI-only signing; do not commit certificates or credentials.

Verification:

- `codesign --verify --deep --strict --verbose=2 "Clawtributor Status.app"`
- `spctl --assess --type execute --verbose "Clawtributor Status.app"`
- Confirm DMG opens and app launches on a clean Mac.
- Confirm Gatekeeper does not show the unidentified developer warning.

## Phase 2: Windows Signing

Preferred path:

- Use Microsoft Trusted Signing / Azure Artifact Signing for cloud-based signing.

Fallback path:

- Use a standard Authenticode code-signing certificate exported as `.pfx`.

GitHub Actions secrets depend on chosen path.

For certificate-based signing:

- `WIN_CSC_LINK`: base64-encoded `.pfx` certificate or secure URL.
- `WIN_CSC_KEY_PASSWORD`: password for the `.pfx`.

For Trusted Signing:

- Add a custom signing hook for electron-builder.
- Store Azure tenant/client/subscription/account/profile identifiers as GitHub secrets.
- Use OIDC or a scoped service principal rather than long-lived broad credentials.

Implementation:

- Change Windows auto-update target to `nsis`.
- Keep Windows ZIP as a manual portable fallback.
- Ensure installer and executable are signed.
- Add timestamping.

Verification:

- Verify Authenticode signature on installer and executable.
- Install on a clean Windows VM.
- Confirm SmartScreen behavior is acceptable for a new publisher.
- Confirm app launches after installation.

## Phase 3: CI Release Workflow

Update `.github/workflows/release.yml`:

- Build on `macos-latest` for macOS artifacts.
- Build on `windows-latest` for Windows artifacts.
- Upload only app artifacts and updater metadata:
  - `*.dmg`
  - `*.zip`
  - `*.exe`
  - `latest.yml`
  - `latest-mac.yml`
- Keep release metadata generated by electron-builder.
- Do not upload unpacked directories, debug YAML, or blockmaps unless the updater requires them.

Expected release assets:

- `Clawtributor.Status-<version>-mac-arm64.dmg`
- `Clawtributor.Status-<version>-mac-arm64.zip`
- `Clawtributor.Status-<version>-mac-x64.zip`
- `Clawtributor.Status-<version>-win-x64.exe`
- `Clawtributor.Status-<version>-win-arm64.exe`
- `latest-mac.yml`
- `latest.yml`

## Phase 4: Auto-Update Implementation

Dependencies:

- Add `electron-updater`.

Main-process behavior:

- Set `autoUpdater.autoDownload = false`.
- Set `autoUpdater.autoInstallOnAppQuit = false`.
- Check once shortly after startup, not immediately.
- Add a manual `Check for Updates...` menu item.
- Never use the user's GitHub OAuth token for update checks.
- Use public GitHub Releases only.

Renderer UX:

- Show nothing when there is no update.
- If an update is available, show a small non-blocking banner.
- User chooses when to download.
- After download, show `Restart to update`.
- If update fails, show a brief error and keep the app usable.

IPC surface:

- `updates:check`
- `updates:download`
- `updates:install`
- `updates:get-state`

States:

- `idle`
- `checking`
- `available`
- `not-available`
- `downloading`
- `downloaded`
- `error`

## Phase 5: Privacy and Trust Controls

- No telemetry by default.
- No update checks tied to GitHub login.
- Do not send contribution data to any server.
- Add a visible local-only data statement in the app.
- Add `Delete local token` action.
- Store GitHub OAuth tokens in Keychain on macOS and Credential Manager on Windows.
- Document exact update endpoints and release asset sources.

## Phase 6: Rollout Strategy

- First signed release: publish without auto-update.
- Second signed release: add manual update check only.
- Third signed release: add background check with manual download.
- Consider staged rollout only after at least one successful signed auto-update.

## Open Decisions

- Whether to use Apple ID app-specific password or App Store Connect API key for notarization.
- Whether to use Microsoft Trusted Signing or a traditional code-signing certificate.
- Whether Windows ARM64 needs first-class installer support now or just ZIP fallback.
- Whether to keep macOS x64 ZIP long term.
- Whether to support beta channel releases.
